## 功能模块

- 🗣️ **口语练习**：仿雅思口语，每天一个话题，支持录音、AI 追问、自定义题库

- 📓 **生词本**：一键添加，支持导出 CSV
- 🧩 **语法解析**：在任意页面选中一段日语文本右键，AI 自动分析语法点/翻译/用法（见下方「语法解析功能」）

`index.html`（仓库根目录）是访问网站前的一道邀请码验证页，替代原来直接访问 `日语学习.html`：

- **邀请码维护**：只改 `config.json` 的 `validCodes` 数组即可（不用碰 `index.html`），改完照常双击同步脚本上线，也可以直接告诉 Claude 加/删谁的码
- **生成规则**：一人一码（不共用），方便单独吊销；格式随意（好记的昵称+数字，或纯随机字符串都行）。**不要把姓名/联系方式写进 `config.json`**——它是公开文件，见下面的局限说明；码和人的对应关系记在 Notion 客户管理表里（见「下一步任务」）
- 目前 `config.json` 里的码（`LEARN2024A/B/C`、`STUDY001/002`）还是占位测试码，正式发给人之前要换成真实码
- **「是否使用」没法自动记录**：纯静态站没有后端，网站没有任何机制能知道某个码被谁在什么时候验证通过，这个状态只能人工在 Notion 表里手动更新
- **已知局限（不是真正的安全门禁）**：验证逻辑完全在前端 JS 里跑，`config.json` 本身请求一下就能看到明文邀请码列表；而且 `日语学习.html` 这个直接地址本身没有任何保护，知道的人可以跳过门禁页直接访问。这只是挡住"随手转发链接被陌生人刷到"的软性门槛，不是访问控制机制。**真正加固的两个方案**（2026-07-14 讨论过，用户决定暂不做，之后主动提起再捡起来）：① 轻量：`config.json` 存邀请码的哈希值而不是明文，验证时比对哈希（挡住随手打开文件看的人，但挡不住会改 DevTools 里 JS 的人）；② 彻底：邀请码校验挪到 EdgeOne Pages Functions（服务端），码完全不下发给浏览器（真正安全，但要多写一个后端函数）。

## 英语跟读模块（2026-07-12 新增）

结构和「跟读练习」完全一样（日期折叠 → 点开有音频 + 逐段原文/翻译 + 生词 + 时间戳跳转），但英语用的是**整段**划分，不是逐句拆分——因为英文缩写、省略号、引号对话按标点强行切分很容易切错，和中文译文对不上（第一次上线时就踩过这个坑，已修复）。

**代码结构**：`日语学习.html` 里 `LISTENING_KINDS` 配置对象让「跟读练习」和「英语跟读」共用同一套渲染引擎（`renderListening`/`loadNextWeek`/`buildListeningEntry` 等函数都带 `kind: 'jp' | 'en'` 参数），没有复制一份代码。数据分别存在 `PRESET_LISTENING`（日语 50 篇）和 `PRESET_LISTENING_EN`（英语，当前 10 篇）两个数组里，浏览器本地存储对应 `state.listening` / `state.listeningEn`。

**已收录 10 篇**（音频文件夹 `E01.xxx` ~ `E10.xxx`，E 前缀避免和日语的 `01`~`50` 文件夹冲突）：
1. 真正厉害的人都戒掉了玻璃心　2. 真正拉开你与周围人的差距　3. 怎样走出负面情绪　4. 瑜伽如何作用于身体和大脑　5. 运动改变大脑　6. 运气对成功有多重要　7. 再艰难也要跳出舒适圈　8. 如何才能幸福　9. 怎样说话人们才爱听　10. 如何在压力下保持冷静

**如果以后还要补充英语讲稿**（流程仅供参考，Claude 每次开新会话需要重新搭这套脚本，没有存在仓库里）：
1. 需要 `pdftotext -bbox`（poppler，`brew install poppler`）按文字的 x/y 坐标位置切分「正文」和「生词侧栏」两栏——不能用普通文本提取或 `-layout`，因为这批 PDF 是双栏排版（正文＋生词侧栏），普通提取容易把生词词条渗进正文中间。每份 PDF 的分栏临界 x 坐标不固定，需要按文档自动检测（取 x>250 之后第一个宽度 ≥5pt 的间隙）。
2. PDF 抽取出来的中日文字符有的是「部首变体」编码（比如「人」被编码成部首符而非正常汉字），需要 Unicode NFKC normalize，个别没有对应关系的部首（齿风骨马门）要手动替换。
3. 音频用 `ffmpeg -ac 1 -b:a 64k -ar 44100` 转成单声道 64kbps 的 audio.m4a（这批是完整长度演讲 10-20 分钟，不是日语那种 1-2 分钟片段，所以体积会大一些，每篇 2-8MB）。
4. 原始 PDF/MP3 素材文件夹（如 `10篇/`）已加入 `.gitignore`，不会被推送上网站，只有转换后的 `E0N.xxx/audio.m4a` 会同步。

## 语法解析功能（2026-07-13 新增）

网页右侧固定三张卡片：🔍 翻译 / 📓 生词本 / 🧩 语法（`.side-panel` 用 `position: sticky` 固定在右上角，随内容滚动跟着走，移动端窄屏自动取消 sticky）。

**语法卡片用法**：在页面任意位置（跟读练习的日文段落、翻译结果等）选中一段日语文本后右键，会拦截默认右键菜单，把选中文本（连同能抓到的中文对照，比如所在段落的 `.cn` 译文）发给 AI，返回「语法点 / 翻译 / 怎么使用（含例句）」，渲染在语法卡片里，也能一键加入生词本。触发逻辑在 `日语学习.html` 的 `document.addEventListener('contextmenu', ...)` + `analyzeGrammar()`。

**AI 供应商链路**：新增 `callDeepseek()` / `callZhipu()`（OpenAI 兼容的 chat completions 接口），和已有的 `callGroq()` / `callClaude()` / `callGemini()` 一起被 `callAIChain()` 统一调用，按「DeepSeek → 智谱 → Groq → Claude → Gemini」顺序尝试第一个配置了 Key 的供应商（国内直连优先）。中→日翻译工具（`translateText()`）也改用了这条统一链路，不再各写一遍。Key 输入框在 ⚙️ 设置弹窗里，存在 `state.settings.deepseekKey` / `zhipuKey`。

**顺带修的 bug**：跟读练习（日语）逐句拆分对照时，如果某段日文和中文按句号拆出来的句子数不一致（译者合并/拆分了句子），旧逻辑会把整段中文塞进第一句、后面几句译文变空白，出现"日语和中文翻译没对上"的现象。检测发现 1-10 篇里 `ted_02/03/06/07/08` 都有这个问题。修法和英语跟读模块当初踩的坑一样：句子数对不上时就不再逐句拆，整段保留成一条（`autoSplitParagraphs()`），保证译文永远和原文对应，代价是那几段可点击跳转的粒度变粗。**已加载过跟读内容的用户**需要点一下跟读练习页面里的「🔄 更新讲稿」按钮才会应用到本地已保存的数据（新加载的篇目自动生效）。
